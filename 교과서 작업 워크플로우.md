Gemini 제안은 **70% 채택, 30% 수정**이 맞습니다.

핵심 진단인 **Over-validation, 컨텍스트 파편화, 4중 리뷰 축소**는 맞습니다. 다만 Gemini 안처럼 “박사과정 멘토 ↔ HTML 작업자” 2회 핑퐁만으로 끝내면, 가장 중요한 **원문 충실성 감사(Source Fidelity Audit)**가 약해집니다. 박사과정 멘토 프롬프트는 “무엇이 중요한가”를 잘 보지만, “PDF의 특정 수식·수치·예시가 원문과 정확히 맞는가”를 전담하도록 설계된 프롬프트는 아닙니다. 따라서 최종 워크플로우는 **2세션 핑퐁을 기본으로 하되, Gemini Deep Research/Pro를 ‘원문 대조 감사관’으로 1회만 끼우는 3축 구조**가 최적입니다.

# Phase 1. 교과서 html 작업자 — Step 1 초안 생성

## 어디서

**Claude Project 또는 ChatGPT Custom GPT/Project** 중 현재 `교과서 html 정리작업자 프롬프트`가 설치된 세션.

개인적으로는 **Claude Opus 4.7 또는 Claude Sonnet 계열의 extended thinking 세션**을 1순위로 권합니다. Anthropic은 Claude Opus 4.7을 장시간 추론과 조사성 작업에 최적화된 모델로 설명하고 있고, extended thinking 기능도 공식 문서에서 제공됩니다. 긴 PDF 기반 구조화 작업에는 이 계열이 잘 맞습니다. ([Anthropic][1])

## 업무 형태

* 일반 채팅 + 파일 첨부
* 가능하면 Extended Thinking / Thinking on
* Deep Research는 사용하지 않음
  이유: 이 단계는 웹 리서치가 아니라 **첨부 PDF 내부 컴파일**입니다.

## 첨부

* 대상 PDF
* 가능하면 페이지 범위 메모
* 이전 세션 Self-Test SR-flag가 있다면 같이 첨부

## 입력 프롬프트

```text
첨부한 PDF의 아래 범위에 대해 Step 1만 실행해주세요.

대상 범위:
- 문헌: [문헌명]
- 챕터/섹션: [Ch.__ / Section __]
- 페이지: p.__–p.__

중요 지시:
1. HTML은 아직 생성하지 말고 Step 1 내용만 생성하세요.
2. 원문에 없는 수치, 예시, 페이지 인용, 실험 조건은 만들지 마세요.
3. PDF에서 수식·표·그림·NONMEM 코드가 깨져 보이는 경우, 맥락상 복원하되 [복원] 또는 [확인 필요]를 표시하세요.

6. 특히 Big Idea, Structural Necessity, Assumptions, Limitations, Confusion Pair, Self-Test, Meta-frame을 빠뜨리지 마세요.

출력은 Step 1 완료 지침까지 포함해 주세요.
```

```text
Step 1 Draft v0
```

# Phase 7. HTML QA 최종 검수

## 어디서

**GPT-5.5 Thinking** 또는 **Claude Opus 4.7**

* 내용 누락/의미 변형 검수: GPT-5.5 Thinking 추천
* 긴 HTML 구조/UX/코드 안정성 검수: Claude Opus 4.7 추천

OpenAI의 Deep Research는 파일·웹·앱을 사용한 복합 리서치 보고서에 적합하지만, 이 QA 단계는 웹 리서치가 아니라 HTML과 Content Lock 대조이므로 일반 GPT-5.5 Thinking이 더 적절합니다. ([OpenAI Help Center][4])

## 업무 형태

* 일반 파일 첨부
* Deep Research 사용하지 않음

## 첨부

1. Content Lock v2
2. Draft HTML v1

## 입력 프롬프트

```text
당신은 HTML QA Auditor입니다.

아래 두 자료를 비교 검수하세요.

자료:
1. Content Lock v2
2. Draft HTML v1

목표:
HTML을 새로 작성하지 말고, 오류와 수정 지점만 찾아주세요.

검수 항목:

## 1. 내용 누락
- Content Lock v2에는 있는데 HTML에서 빠진 내용

## 2. 의미 변형
- HTML 변환 과정에서 원래 의미가 바뀐 문장

## 3. MathJax / LaTeX 위험
- 렌더링 깨짐 가능성이 있는 수식
- escape 문제
- inline/display math 혼용 문제

## 4. 코드블록 / 표 / 아코디언 문제
- 코드 포맷 깨짐
- 표가 모바일에서 깨질 위험
- 아코디언 열림/닫힘 구조 문제

## 5. 학습 UX 문제
- 카드가 너무 긴 부분
- 내비게이션이 불명확한 부분
- Self-Test 사용성이 떨어지는 부분

## 6. 수정 우선순위
아래 형식으로 정리하세요.
- Critical: 반드시 수정
- Major: 수정 권장
- Minor: 선택 수정

주의:
1. 새 내용을 보강하지 마세요.
2. 원문에 없는 설명을 추가하지 마세요.
3. HTML 전체를 다시 쓰지 말고, 수정 위치와 수정 지시만 주세요.
```

## 산출물

```text
HTML QA Report v1
```

그 다음 HTML 작업자 세션에 돌아가서:

```text
아래 HTML QA Report v1의 Critical과 Major 항목만 반영하여 Draft HTML v1을 수정하세요.
새 내용은 추가하지 말고, 렌더링/구조/누락/의미 변형만 수정하세요.

[HTML QA Report v1 붙여넣기]
```

## 최종 산출물

```text
Final HTML
Final Markdown Content Lock v2
Audit Log
```

# 시간이 부족할 때 쓰는 초압축 버전

매번 풀버전을 돌릴 필요는 없습니다. 챕터 난이도에 따라 나누세요.

## A급 챕터

예: clearance, Vd, hepatic extraction, compartment model, nonlinear PK, NLME, FOCE, residual error, covariate model

```text
Step 1 → Gemini Source Audit → Mentor Crucible → GPT Content Lock → Step 2 → HTML QA
```

## B급 챕터

예: 비교적 설명적인 개요 챕터, 반복 개념이 많은 부분

```text
Step 1 → Mentor Crucible → GPT Content Lock → Step 2
```

## C급 챕터

예: 이미 잘 아는 내용, 복습용 보조 자료

```text
Step 1 → Step 2 → HTML QA만 간단히
```

# 최종 권고

Gemini의 “핑퐁 프로토콜”은 방향은 좋지만 **너무 과감하게 줄였습니다.** 최종안은 다음 한 문장으로 정리됩니다.

> **외부 일반 리뷰는 모두 제거하되, Gemini는 ‘Source Fidelity Audit’으로 1회만 남기고, 박사과정 멘토는 ‘P1/P3 Crucible Review’로만 사용하며, GPT-5.5 Thinking이 모든 피드백을 Content Lock으로 통합한 뒤 HTML화한다.**

이 방식이 현재 프롬프트 자산을 가장 잘 살리면서도, 시간 낭비와 검증 과잉을 동시에 줄이는 구조입니다.
