## §2 — 개념 해부 카드

### 카드 C-1: 데이터셋 구조 원칙

---

#### ━━ [개념 Big Idea: 거장의 시각] ━━

**왜 치명적으로 중요한가:** 투약 레코드의 TIME이 관측 레코드보다 1초라도 뒤에 놓이거나, 동일 레코드에 AMT와 DV가 함께 존재하면 — NONMEM은 에러를 내거나 더 나쁜 경우, 조용히 잘못된 PK 파라미터를 추정한다. NDA 제출 직전 데이터셋 오류가 발견된 사례는 드물지 않다.

**체화해야 할 단 하나의 직관:** 데이터셋은 "행 = 한 순간에 일어난 단 하나의 사건"이라는 원칙 위에 서 있다. 투약과 관측은 절대 같은 행에 공존할 수 없다 — 현실에서 동시에 일어나더라도 두 행으로 분리한다.

**이 직관을 머릿속에 박고 아래를 읽어라:** 모든 규칙이 이 한 가지 원칙의 파생임을 기억하라.

---

**A. Formal Definition**

NONMEM(NM-TRAN) 분석 데이터셋은 **ASCII 형식 평면 파일(flat file)**로, 행(레코드)과 열(데이터 항목)의 2차원 배열이다. 각 레코드는 특정 시점에 동시적으로 발생한 사건의 집합이다. PREDPP를 사용하는 모든 모델은 데이터를 **시간순 이벤트(time-ordered events)**로 처리하며, 레코드는 세 유형으로 분류된다:

- **투약 이벤트(Dosing event):** AMT ≠ 0, DV = missing
- **관측 이벤트(Observation event):** DV ≠ missing, AMT = missing
- **기타 유형 이벤트(Other-type event):** 공변량 갱신, 시스템 재설정 등

**B. Derivation / Code Structure**

PREDPP를 사용하는 제어구문의 최소 필수 3행:

```fortran
$PROBLEM HO_IV_1comp          ; 분석 제목 — 모델 버전 관리의 출발점
$DATA ../HOdata_IV.csv IGNORE=@ ; 파일 경로 + @ 시작 행(헤더) 제거
$INPUT ID TIME AMT DV MDV SEX AGE WT HT ; 열 순서 선언 — 데이터셋 열과 1:1 대응
```
`[출처: PIPET 2장, p.14–15]`

올바른 데이터셋 예시 (Owen Table 4.4 기반):

```
; 투약과 관측이 동일 시점에 발생할 경우 — 반드시 두 행으로 분리
ID  TIME  AMT   DV    MDV
001  0.0   100   .     1    ← 투약 레코드: DV 결측
001  0.0   .     35.4  0    ← 관측 레코드: AMT 결측
001  0.5   .     12.6  0
```

잘못된 코딩 — **절대 금지:**

```
; ❌ 동일 레코드에 AMT=100, DV=35.4 공존 → NM-TRAN 에러 또는 무결성 훼손
ID  TIME  AMT   DV
001  0.0   100   35.4   ← NONMEM이 거부하거나 예측 불능 상태
```
`[출처: Owen Ch.4, Table 4.3–4.4, p.70]`

**C. Structural Necessity**

이 이분법(dosing/observation 분리)이 불가피한 이유: PREDPP는 내부적으로 **구획 모델의 ODE를 시간 축을 따라 전진적분(forward integration)**한다. 적분기는 한 레코드를 처리할 때 "지금 약이 들어왔는가(AMT), 아니면 측정이 이루어졌는가(DV)"를 분기해야 한다. 두 정보가 동일 레코드에 있으면 분기 자체가 불가능해진다.

**D. Assumptions & Boundary Conditions**

- 모든 레코드는 동일한 열 수를 유지해야 한다. 누락값은 `"."` 또는 `"0"`으로 채운다. 빈칸은 허용되지 않는다(space-delimited 파일에서).
- 각 개체의 레코드는 **연속적으로(contiguous)** 배열되어야 한다. ID가 섞이면 NONMEM은 분리된 ID를 다른 개체로 처리한다.
- TIME은 개체 내에서 단조 증가(non-decreasing)해야 한다. 단, 재설정 이벤트(EVID=3/4)가 있을 경우 예외.

**E. Limitations**

- NONMEM 7 기준 한 번의 실행에서 처리 가능한 최대 데이터 항목 수: **50개**. 초과 시 `DROP` 옵션 활용 필수.
- 스프레드시트(Excel) 기반 데이터셋 생성은 21 CFR Part 11 규제 요건(변경 이력 추적)을 충족할 수 없어 Phase 2 이상의 규제 제출에 부적합하다.
- 시간 형식 혼용(clock time vs. decimal time)은 `ELAPSED TIME MAY NOT BE NEGATIVE` 에러를 발생시킨다.

**F. Five Cognitive Layers**

| 레이어 | 내용 |
|---|---|
| **L1 형식적 규칙** | ASCII flat file, 2D 배열, 레코드 유형 3분류, MDV/AMT/DV 상호 배타성 |
| **L2 구조적 직관** | 데이터셋 = ODE 적분기를 위한 "사건 큐(event queue)" — 투약은 초기조건 변경, 관측은 잔차 계산 트리거 |
| **L3 구조적 동일성** | 항공 관제 비행 계획(비행 이벤트의 시간순 스트림)과 동형: 이륙/착륙/관측이 분리된 이벤트로 관리됨 |
| **L4 생물학적 의미** | 투약 레코드 = 약물이 체내로 진입하는 물리적 순간; 관측 레코드 = 그 결과로 혈중에서 샘플링되는 순간 |
| **L5 실무 투영** | 데이터 오류의 70%는 TIME 오기, AMT/DV 혼용, 비연속 ID 배열에서 발생 — 첫 번째 QC 체크포인트 |

NONMEM PREDPP 데이터셋은 시간순 이벤트 스트림이다. 투약 레코드(AMT ≠ 0, DV = missing)와 관측 레코드(DV ≠ missing, AMT = missing)는 절대 동일 행에 공존할 수 없다 — ODE 적분기의 분기 논리가 이를 요구하기 때문이다. 개체 레코드는 연속 배열(contiguous)되어야 하며, TIME은 개체 내 단조 증가해야 한다. 데이터셋 오류는 추정치를 조용히 오염시킨다.

### 카드 C-2: 예약 변수 체계 ⚡ [Apex Concept]

---

#### ━━ [개념 Big Idea: 거장의 시각] ━━

**왜 치명적으로 중요한가:** NONMEM에 내장된 예약 변수명을 사용자 정의 변수명으로 오버라이드하거나 그 의미를 잘못 이해한 채 데이터셋을 만들면, NONMEM은 에러 없이 실행되지만 완전히 다른 모델을 추정한다 — 예를 들어 EVID의 오기는 특정 관측값을 피팅에서 제외하거나, SS=1 누락은 수백 개의 허구 투약 이벤트를 삽입시킨다.

**체화해야 할 단 하나의 직관:** 예약 변수는 NONMEM과 사용자 사이의 **계약서 조항**이다. 각 변수의 값은 "지금 이 행에서 시스템 상태가 어떻게 변하는지"를 NONMEM에게 전달하는 신호이지, 단순한 숫자가 아니다.

**이 직관을 머릿속에 박고 아래를 읽어라:** 아래 각 변수를 "숫자"가 아닌 "시스템 상태 신호"로 읽으면 규칙이 전부 자연스럽게 도출된다.

---

**A. Formal Definition**

NONMEM(NM-TRAN)에 미리 예약된 데이터 항목 레이블은 사용자가 정의하는 것이 아니라 시스템이 특정 의미로 해석하도록 설계된 변수명이다. 이 변수들을 사용자 정의 목적으로 재사용하는 것은 금지된다.

핵심 예약 변수 전체 목록 (PIPET 표 2.2 + Owen 4.3절 통합):

| 변수 | 상황 | 의미 |
|---|---|---|
| ID | 항상 | 개체 식별자 (숫자, 연속 배열 권장) |
| DV | 항상 | 종속변수 — 피팅 대상 관측값 |
| MDV | 항상 | 결측 종속변수 플래그 (1=결측, 0=존재) |
| TIME | PREDPP | 이벤트 시간 (단조 증가, non-negative) |
| AMT | PREDPP | 투여 용량 (투약 레코드에만, 관측 레코드는 결측) |
| RATE | PREDPP | 투여속도 (정주 시) — 값 해석 별도 |
| ADDL | PREDPP | 추가 투여 횟수 (첫 투여 제외한 N-1) |
| II | PREDPP | 투여 간격 (ADDL/SS와 함께 사용) |
| SS | PREDPP | 항정상태 플래그 (0/1/2) |
| CMT | PREDPP | 구획 지정 (투약·관측 구획 분리 시 필수) |
| EVID | PREDPP | 이벤트 유형 명시적 선언 (0/1/2/3/4) |

`[출처: PIPET Ch.2, 표 2.2, p.19 + Owen Ch.4, §4.3, pp.71-80]`

**B. Derivation / Code Structure — EVID 체계 완전 해부**

EVID는 각 레코드의 "이벤트 유형"을 명시적으로 선언한다. MDV와의 관계는 독립적이되 상호 보완적이다:

```
EVID=0 → 관측 이벤트 (DV 필요, AMT 결측 필수)
EVID=1 → 투약 이벤트 (AMT 필요, DV 결측 필수)
EVID=2 → 기타 유형 이벤트 (공변량 갱신, 예측값 반환용)
EVID=3 → 시스템 리셋 (구획 내 양을 0으로 초기화)
EVID=4 → 리셋 + 즉시 투약 (EVID=3 + EVID=1 조합)
```

EVID별 AMT·DV 조건 요약 (Owen Table 4.9):

```
; 피팅 시 일반 요건
EVID  |  DV                    | AMT
--------------------------------------------------
  0   |  Generally nonmissing  | Required to be missing
  1   |  Required to be missing| Required to be nonmissing
  2   |  Required to be missing| Required to be missing
  3   |  Required to be missing| Required to be missing
  4   |  Required to be missing| Required to be nonmissing
```
`[출처: Owen Ch.4, Table 4.9, p.76]`

**MDV와 EVID의 분업:**
- MDV: "이 행의 DV를 피팅에 포함시킬 것인가?" — 피팅 포함 여부 제어
- EVID: "이 행은 어떤 종류의 사건인가?" — 시스템 상태 변화 유형 선언
- EVID=2 레코드는 MDV=1이지만, NONMEM이 해당 시점의 **모델 예측값을 반환**하도록 지시한다 — GOF 플롯 구성, 전체 PK 프로파일 재구성에 활용.

**C. Structural Necessity**

EVID 체계가 이 정확한 5-값 구조를 가져야 하는 이유: ODE 적분기는 레코드를 순서대로 처리하면서 각 레코드에서 ①구획 내 약물 양을 변경할 것인지(투약), ②잔차를 계산할 것인지(관측), ③시스템을 초기화할 것인지(리셋)를 결정해야 한다. 이 세 가지 분기 + 그 조합이 정확히 5개의 EVID 값을 구조적으로 필연화한다.

**C-2. Plausible Fallacy (그럴싸한 오류) ← Apex Concept 적용**

**오류의 구체적 형태:**

> "EVID는 어차피 MDV로 대체 가능하다 — DV 결측 행에 MDV=1만 넣어주면 EVID는 생략해도 되고, EVID=2는 그냥 MDV=1로 처리하면 된다."

이 오류는 다음 두 가지 경로로 모델을 손상시킨다:

**나비효과 경로 1 — 교차시험 세션 간 농도 오염:**
교차시험(crossover study)에서 Period 1의 약물이 충분히 소실된 후 Period 2가 시작된다고 가정하자. EVID=3(리셋)을 사용하지 않으면 NONMEM은 Period 1의 잔류 약물 양을 Period 2 적분에 포함시킨다 — 사실상 없는 농도를 더하는 것이다. 결과: Period 2의 CL이 **체계적으로 과소추정**되고, 이 편향이 IIV 추정치로 흡수되어 η-shrinkage가 비정상적으로 증가한다.

**나비효과 경로 2 — EVID=2 예측값 반환 누락:**
PK/PD 동시 피팅에서 PD 관측 레코드와 동일 시점의 PK 예측 농도가 필요할 때, EVID=2 레코드를 MDV=1로 대체하면 NONMEM은 해당 시점의 예측값을 \$TABLE에 출력하지 않는다. 결과: PD 모델에 공급되는 예측 농도(Cpred)가 비어, PK/PD 링크 모델 전체가 붕괴된다.

**이 오류를 발견하는 GOF 시그니처 패턴:**

> **"Cross-Period Phantom Elevation" 시그니처:** CWRES vs. TIME 플롯에서 각 교차시험 Period의 초반부(0–2×t½ 구간)에 **양의 방향 편향 클러스터**가 반복적으로 나타난다. 직관적으로는 "흡수 모델 문제"로 오해하기 쉽지만, 실제 원인은 이전 Period 잔류 약물의 잘못된 중첩이다.

**D. Assumptions & Boundary Conditions**

- MDV는 PREDPP에서 필수가 아니지만, 명시적으로 정의하지 않으면 NONMEM이 맥락에서 추론한다 — 이 추론이 사용자 의도와 다를 수 있으므로 항상 명시 권장.
- EVID도 필수가 아니지만 교차시험·다기간 데이터·PK/PD 동시 피팅 시에는 필수적으로 명시해야 한다.
- CMT는 ADVAN1/3/11처럼 기본 투약·관측 구획이 정의된 모델에서는 생략 가능하지만, ADVAN2/4에서 IV loading dose + 경구 유지요법 혼용 시 **반드시** 명시해야 한다 (IV 볼루스 → CMT=2, 경구 → CMT=1).

**E. Limitations**

- NONMEM 7의 ID 필드는 최대 14자리를 인식하지만 \$TABLE 기본 출력 형식(s1PE11.4)은 5자리만 표현한다 → 다기관 병합 데이터셋에서 10126(연구10, 환자126)과 10127이 동일 ID로 표출될 수 있다. 해결: RFORMAT 옵션 사용.
- PCMT 항목은 EVID=1/2 레코드에서만 작동하며, EVID=0 관측 레코드에서는 CMT가 PCMT를 결정한다.

**F. Five Cognitive Layers**

| 레이어 | 내용 |
|---|---|
| **L1 형식적 규칙** | 예약 변수 11종의 완전한 정의, EVID 0–4의 AMT/DV 조건 테이블 |
| **L2 기하학적 직관** | 각 변수는 ODE 적분기의 상태공간(state space)에서 특정 차원을 제어하는 스위치 |
| **L3 구조적 동일성** | 항공 이벤트 로그: EVID = 이벤트 유형 코드(이륙/착륙/비상), MDV = 해당 이벤트 기록 여부 |

| **L5 실무 투영** | 규제 제출 시 데이터셋 명세서(dataset specification document)에 각 예약 변수의 사용 근거 명시 필수 |

### 카드 C-3: 투여 모델링 고급 기법 — ADDL/II·SS·RATE

---

#### ━━ [개념 Big Idea: 거장의 시각] ━━

**왜 치명적으로 중요한가:** Phase 3 임상 데이터에서 환자가 수개월간 하루 두 번 복용했다면, 각 투약을 개별 레코드로 작성하면 수백 줄의 레코드가 된다. ADDL/II를 잘못 이해해 총 투여 횟수(N)를 ADDL에 입력하면 약 하나가 더 투여된다 — 이 오류는 CL을 체계적으로 과소추정시킨다.

**체화해야 할 단 하나의 직관:** ADDL은 "추가로 더 투여한 횟수"이지 "총 투여 횟수"가 아니다 — ADDL = N−1.

**이 직관을 머릿속에 박고 아래를 읽어라:** 아래 SS·RATE의 값 체계도 전부 "첫 이벤트는 레코드에 명시, 나머지는 묵시적 선언"이라는 같은 원칙으로 작동한다.

---

**A. Formal Definition**

복잡한 투약 이력을 최소한의 레코드로 표현하기 위한 세 가지 도구:

1. **ADDL + II**: 동일 용량을 일정 간격으로 반복 투여 시 압축 표현
2. **SS (=0/1/2)**: 항정상태 여부 및 이전 투약 이력 처리 방식 선언
3. **RATE (=0/양수/−1/−2)**: 정주 투여 속도 또는 투여 지속시간 지정

**B. Derivation / Code Structure**

**ADDL + II 예시** (Owen Table 4.10 + PIPET 그림 2.7 기반):

```
; 경구 항생제 250mg, 하루 2회 10일간 → 총 20회 투여
; Day 1(0시간)·Day 10(220시간) 채혈
ID    TIME  AMT  ADDL  II   DV    EVID  MDV  CMT  TAD
1     0     250  19    12   .     1     1    1    0
1     1.0   .    .     .    10.6  0     0    2    1.0
1     220   .    .     .    12.3  0     0    2    4.0

; ADDL=19 해석:
; TIME=0에 첫 번째 투약(명시) + 이후 12시간 간격으로 19회 추가(묵시)
; = 총 20회. ADDL = 20 - 1 = 19
```
`[출처: Owen Ch.4, Table 4.15, p.82]`

**SS 값 체계** (Owen §4.3.12 + PIPET §2.5.9):

```
SS=0: 항정상태 아님 — 이전 투약 이력 그대로 유지
SS=1: 항정상태 → 구획 내 양을 이론적 SS 조건으로 재설정
      이전 투약 레코드 효과는 무시(reset). II 반드시 명시.
SS=2: 항정상태 + 이전 투약 이력 유지 (superposition)
      → 예정에 없던 추가 투여가 있었을 때 사용
```

**SS=2 적용 예시** — 만성 투여 중 비정기 추가 투여 (Owen Table 4.18):

```
; 매일 7:00AM 250mg 만성 투여 환자 + 11/20 18:00 추가 투여
; 11/22 10:30 채혈
ID  DATE      TIME   AMT  SS  II  DV    EVID  MDV  CMT  TAD
1   11/20/10  18:00  250  0   .   .     1     1    1    0    ← 추가 투여 (비항정상태)
1   11/22/10  7:00   250  2   24  .     1     1    1    0    ← 만성 SS 투여 (superposition)
1   11/22/10  10:30  .    .   .   23.4  0     0    2    3.5
```
`[출처: Owen Ch.4, Table 4.18, p.84]`

**RATE 값 체계** (PIPET §2.5.7 + Owen §4.3.9):

```
RATE = 양수  → 직접 투여 속도 지정 (단위: AMT/TIME 단위)
               지속시간 = AMT / RATE
RATE = 0     → 볼루스 투여 (정주입이 아님)
RATE = -1    → 투여속도를 파라미터로 추정 → $PK에 R1=THETA(n) 필요
RATE = -2    → 투여지속시간을 파라미터로 추정 → $PK에 D1=THETA(n) 필요
```

RATE=−2 실제 예시 — 경피 흡수 제제에서 흡수 지속시간 추정:

```fortran
$INPUT ID TIME AMT DUR RATE DV EVID MDV CMT
$PK
  D1 = DUR    ; 데이터셋의 DUR 열을 투여 지속시간으로 사용
              ; 또는 D1=THETA(1)로 추정 가능
```
`[출처: Owen Ch.4, §4.5.2 + PIPET §2.5.7, p.22-23]`

**불규칙 투여 간격에서의 창의적 SS 코딩** (Owen Table 4.19 — 하루 3회 불규칙 스케줄):

```
; 매일 8:00, 14:00, 20:00 10mg → 각 시간대를 24시간 1회 regimen 3개로 분해
ID  DATE  TIME   AMT  SS  II  EVID  MDV  CMT  TAD
1   13    14:00  10   1   24  1     1    1    0   ← 첫 번째 regimen, SS=1
1   13    20:00  10   2   24  1     1    1    0   ← 두 번째 regimen, superposition
1   14    8:00   10   2   24  1     1    1    0   ← 세 번째 regimen, superposition
1   14    8:30   .    .   .   0     0    2    0.0 ← 채혈
```
`[출처: Owen Ch.4, Table 4.19, p.85]`

**C. Structural Necessity**

ADDL/SS 체계가 단순히 "편의 기능"이 아닌 구조적 필연인 이유: PopPK 분석은 수십~수백 명 환자의 수개월 투약 이력을 처리해야 한다. 모든 투약을 개별 레코드로 나열하면 데이터셋 크기가 기하급수적으로 증가하고, NM-TRAN 처리 시간이 늘어나며, 수동 입력 오류 위험이 폭증한다. ADDL/SS는 이 문제를 수학적 압축으로 해결한다.

**D. Assumptions & Boundary Conditions**

- ADDL 사용 시 II가 일정해야 한다. 불규칙 간격은 여러 ADDL 레코드로 분해 또는 개별 레코드로 대체.
- 묵시적 ADDL 투여 레코드에서 공변량 값은 **다음 물리적 레코드**의 값을 사용한다 (이전 레코드 값이 아님) — 체중이 중간에 변한 경우 의도치 않은 조기 반영 가능. \$BIND로 제어 가능.
- SS=1 사용 시 II 누락은 NM-TRAN 에러.

**E. Limitations**

- SS 구현은 투약 이력이 충분히 길어 실제 항정상태에 도달했다는 가정 하에서만 유효하다. t½ × 5 미만의 투약 기간에서 SS=1 사용은 구획 내 양을 과대추정한다.
- ADDL로 지정된 묵시적 투여 레코드 중 관측과 동일 시점이 겹칠 경우 NONMEM은 묵시적 투약이 먼저 발생한 것으로 처리한다 → 사전 채혈(pre-dose trough) 시각 처리 시 8.001h처럼 미세 시간 조정 필요.

**F. Five Cognitive Layers**

| 레이어 | 내용 |
|---|---|
| **L1 형식적 규칙** | ADDL=N−1, II 단위 통일, SS=0/1/2 정의, RATE 4가지 값 완전 명세 |
| **L2 구조적 직관** | ADDL = "반복 패턴의 압축 표현" — 무한 반복을 유한 선언으로 접음 |
| **L3 구조적 동일성** | 정기 구독 서비스(SS=1: 과거 모든 배송 이력 포함한 최신 상태)와 동형 |
| **L4 생물학적 의미** | SS=1: 5×t½ 이상 복용 후 체내 약물 양이 이론적 정상상태에 도달한 생리적 사실을 코드로 선언 |
| **L5 실무 투영** | 다기관 Phase 3 데이터에서 투약 이력 재구성은 SAS/R 프로그램으로 자동화 — 스프레드시트 수작업 금지 |

### 카드 C-4: 제어구문 연결 — \$PROB·\$DATA·\$INPUT

---

#### ━━ [개념 Big Idea: 거장의 시각] ━━

**왜 치명적으로 중요한가:** \$INPUT의 열 선언이 데이터셋 실제 열 순서와 어긋나면 — 예를 들어 데이터셋에서 AMT가 3번째 열인데 \$INPUT에 4번째로 선언하면 — NONMEM은 에러 없이 실행하지만 잘못된 열을 AMT로 읽는다. 이 오류는 출력 파일에서는 거의 보이지 않는다.

**체화해야 할 단 하나의 직관:** \$INPUT은 데이터셋 열의 **"이름표(label)"를 순서대로 붙이는 선언**이다 — 데이터셋 자체는 바뀌지 않는다.

**이 직관을 머릿속에 박고 아래를 읽어라:** \$DATA의 IGNORE/ACCEPT는 데이터셋을 필터링하고, \$INPUT은 열에 이름을 부여하며, \$PROB는 실행 이력을 관리한다 — 세 레코드의 역할은 완전히 분리되어 있다.

---

**B. Derivation / Code Structure**

```fortran
;─────────────────────────────────────────────────────
; 제어구문 파일 필수 3행 완전 예시
;─────────────────────────────────────────────────────

$PROBLEM HO_IV_1comp_BaseModel_v1
; ↑ 버전 관리 핵심 — 모델 복잡도 증가 시 v2, v3로 변경

$DATA ../HOdata_IV.csv IGNORE=@
; ↑ IGNORE=@: 첫 글자가 알파벳(@)인 행 제거 = 헤더 제거
; 대안: 데이터셋 첫 열에 #ID → 제어구문에 IGNORE=# 사용
; ACCEPT 예시: IGNORE=(SEX.EQ.1) → 남성(SEX=1)만 제외
; IGNORE=(AGE.GE.65) → 65세 이상 제외

$INPUT ID TIME AMT DV MDV SEX AGE WT HT
; ↑ 데이터셋 열 순서와 반드시 1:1 일치
; 불필요 열: 변수명=DROP (예: RACE=DROP)
; 변수명 재매핑: AMT=DOSE (데이터셋의 'DOSE' 열을 AMT로 인식)
```
`[출처: PIPET Ch.2, §2.1, pp.13-15 + Owen Ch.4, §4.2, p.69]`

**단위 통일 전략 (AMT-DV 단위 불일치 해결):**

```fortran
; 방법 1: 데이터셋에서 미리 변환 (AMT: mg → ng)
; 데이터셋: AMT = 100000 (ng), DV = 891.2 (ng/mL) ← PIPET 표 2.1 실제 예시

; 방법 2: 제어구문의 $PK에서 scaling factor 적용
$PK
  S1 = V/1000   ; AMT(mg) → 농도(ng/mL) 변환 시
                ; [AMT(mg) / V(L)] × 1000 → µg/L → × 1 → µg/L 아님
                ; 정확: S1 = V (L) → C = A1/S1 = mg/L = µg/mL ×1000
```
`[출처: PIPET Ch.2, §2.5.6, p.22]`

```yaml

\$PROB·\$DATA·\$INPUT은 제어구문 파일 첫 세 줄로 각 역할이 완전 분리된다. \$DATA의 IGNORE=@은 알파벳으로 시작하는 헤더 행을 제거하며, \$INPUT은 데이터셋 열 순서에 이름을 부여한다 — 열 선언 오류는 에러 없이 잘못된 파라미터를 산출한다. AMT와 DV의 단위 불일치는 데이터셋 선처리(사전 변환) 또는 \$PK의 scaling factor(S1=V 등)로 해결한다.

### 카드 C-5: 공변량·시변 데이터·헤더 처리

---

#### ━━ [개념 Big Idea: 거장의 시각] ━━

**왜 치명적으로 중요한가:** 장기 연구에서 체중이 변화할 때 ADDL로 지정된 묵시적 투약 레코드의 공변량 값은 이전 레코드가 아닌 **다음 물리적 레코드**의 값을 사용한다 — 이를 모르면 체중 기반 공변량 모델이 의도와 다른 시점에서 체중을 바꾸게 된다.

**체화해야 할 단 하나의 직관:** NONMEM은 묵시적 이벤트의 공변량을 "미래로부터" 가져온다.

**이 직관을 머릿속에 박고 아래를 읽어라:** 아래 \$BIND 언급은 이 기본값을 제어하는 고급 기법이다.

## §5 — Confusion Pair Dissection

### 혼동쌍 1: MDV vs. EVID

| 비교 차원 | MDV | EVID |
|---|---|---|
| **표면적 유사성** | 둘 다 "이 레코드가 관측값을 포함하는가"를 묻는 것처럼 보인다 | |
| **수식/코드 형태** | `MDV = 0` (관측 포함) / `MDV = 1` (관측 제외) | `EVID = 0` (관측) / `1` (투약) / `2` (기타) / `3` (리셋) / `4` (리셋+투약) |
| **생물학적·구조적 지시체** | "이 행의 DV를 OFV 계산(피팅)에 포함시킬 것인가?" — **피팅 포함 여부** | "이 행에서 시스템에 어떤 사건이 일어나는가?" — **사건 유형 선언** |
| **변화 방향** | MDV=1이어도 EVID=2이면 모델 예측값이 반환됨 (피팅에는 미포함) | EVID=2는 MDV=1을 내포하지만, MDV=1은 EVID=2를 내포하지 않음 |
| **임상/모델링 결과** | MDV만 사용하고 EVID 생략 시: 교차시험 period 간 농도 중첩, PK/PD 링크 모델 예측값 누락 | |
| **⚡ 기억 고리** | **MDV = 채점표 (이 관측값 점수를 줄 것인가?)·EVID = 출석부 (오늘 어떤 수업이 있었는가?)** — 채점표에 0이어도 수업 종류는 별도로 선언해야 한다 | |

---

### 혼동쌍 2: SS=1 vs. SS=2 ⚡ [Critical Blow 적용]

| 비교 차원 | SS=1 | SS=2 |
|---|---|---|
| **표면적 유사성** | 둘 다 "항정상태 투약"을 선언한다 | |
| **코드 형태** | `SS=1, II=tau` | `SS=2, II=tau` |
| **구조적 지시체** | **이전 투약 이력 모두 무시(reset)** 후 이론적 SS 구획 양으로 재설정 | **이전 투약 이력 유지(superposition)** 후 SS 조건 중첩 |
| **변화 방향** | 교차시험 Period 2 첫 투약 → SS=1로 Period 1 잔류 약물 소거 | 만성 투여 중 비정기 추가 투여 → SS=2로 추가 투여 효과를 만성 SS에 중첩 |
| **임상/모델링 결과** | SS=1을 써야 할 곳에 SS=2를 쓰면: 이전 Period 약물 효과가 현재 Period에 유령처럼 중첩 → CL 과소추정·η 팽창 | |
| **⚡ 기억 고리** | **SS=1은 "백지에서 다시 시작"(reset + SS 계산), SS=2는 "이전 장에 새 이야기를 덧쓰기"(superposition)** — 교차시험이면 백지, 만성 중 추가 투여면 덧쓰기 | |
| **치명적 타격** | 교차시험 데이터에서 SS=1 대신 SS=2를 사용해 NDA 제출하면: 심사관이 Period별 PK 파라미터 비교에서 체계적 편향(systematic bias)을 발견하여 Deficiency Letter를 발행하고, 재분석을 요구한다 — 이미 제출된 용량 권고도 재검토 대상이 된다 |

---

### 혼동쌍 3: ADDL(추가 투여 횟수) vs. N(총 투여 횟수)

| 비교 차원 | ADDL | N (총 횟수) |
|---|---|---|
| **표면적 유사성** | 둘 다 "몇 번 투여했는가"를 나타내는 것처럼 보인다 | |
| **정의** | ADDL = N − 1 (레코드에 명시된 첫 투여를 제외한 추가 횟수) | N = 총 투여 횟수 |
| **실수 시나리오** | N=10 투여에서 ADDL=10 입력 → 실제로 11회 투여가 시뮬레이션됨 | |
| **임상/모델링 결과** | 총 AUC 약 10% 과대추정, CL 과소추정 — 소규모 데이터셋에서 θ(CL) 편향이 η로 흡수 | |
| **⚡ 기억 고리** | **"ADDL은 '추가'다 — 이미 레코드에 한 번이 있으니 거기서 하나 뺀다." ADDL = N − 1은 수학 공식이 아니라 언어 정의다: Additional = 이미 있는 것에 더하는 것** | |

> **문제 B:** 데이터셋을 수정하지 않고 제출하면 23명(전체의 1.8%)의 데이터에 체계적 편향이 존재한다. 민감도 분석에서 이 23명을 제외해도 최종 CL 추정치는 4.2% 차이가 난다.
>
> **질문:** 당신은 수정 후 재분석(일정 3주 지연)을 택하는가, 아니면 민감도 분석 결과를 첨부하고 현재 데이터셋으로 제출하는가?

<details>
<summary>수석 모델러의 Trade-off 논리</summary>

**수석 모델러의 답변:**

**반드시 수정 후 재분석한다.** 이유는 다음과 같다:

**Trade-off 분석:**

*제출 강행의 유혹:* 23명/1247명(1.8%)은 통계적으로 소수다. 4.2% 차이는 CL 추정의 불확실도 범위 내일 수 있다. 3주 지연은 상업적 비용이 크다.

*그러나 반드시 수정해야 하는 세 가지 구조적 이유:*

1. **데이터셋 오류는 "알려진 편향(known bias)"이다.** 불확실성(uncertainty)과 편향(bias)은 근본적으로 다르다. 심사관은 랜덤 변동성을 허용하지만 알려진 체계적 오류는 허용하지 않는다. 심사관이 독립적으로 데이터셋을 재현 분석했을 때 이 오류를 발견하면 Deficiency Letter가 아닌 Complete Response Letter(사실상 거절)로 이어질 수 있다.

2. **ICH E9(R1)과 FDA Population PK 가이던스는 데이터 무결성을 요구한다.** SS 오코딩은 모델이 아닌 데이터 레벨의 오류다 — 모델 선택 불확실성과는 다르다. 민감도 분석은 모델 가정의 변동을 탐색하는 도구이지, 알려진 데이터 오류를 정당화하는 도구가 아니다.

3. **규제 문서에서의 방어 가능성:** 수정 후 재분석 보고서에서는 "오류 발견 → 수정 → 재분석 → 결론 안정성 확인"의 완결된 QC 이력을 제시할 수 있다. 이것이 가장 강력한 규제 방어다.

**규제 문서 표현:** "A dataset error was identified during final QC review: 23 subjects in the crossover cohort had SS=2 coded for the first dose of Period 2, where SS=1 was indicated. The dataset was corrected, and analysis was repeated. The final PK parameter estimates differed by less than 5% from the pre-correction analysis, confirming the robustness of the model."

**결론:** 3주 지연의 단기 비용보다 데이터 무결성 훼손의 장기 규제 리스크가 훨씬 크다.

</details>

```

### B. Dependencies — 이 섹션을 대충 넘겼을 때 고급 모델링에서 발생하는 실패 모드

**실패 모드 1 — "무증상 CL 편향":**
ADDL에 총 투여 횟수(N)를 입력하는 실수(ADDL=N 대신 ADDL=N−1이어야)는 추정된 CL을 5~15% 과소추정시킨다. 이 오류는 OFV에서 감지되지 않고, GOF 플롯에서도 뚜렷한 패턴이 나타나지 않는다 — 잔차가 N개의 허구 투약 이력에 맞게 파라미터가 조정되기 때문이다. 규제 제출 후 심사관의 독립 재현 분석에서 발견되는 시나리오가 실제로 보고되어 있다.

**실패 모드 2 — "PK/PD 링크 단절":**
PK/PD 동시 분석에서 PD 관측 시점의 예측 농도를 반환하기 위한 EVID=2 레코드를 MDV=1로 대체하면, \$TABLE에 해당 시점의 농도 예측이 출력되지 않는다. 이 상태에서 effect compartment 모델을 구현하면 Ce(t) 계산이 불가능해져 PD 피팅 전체가 붕괴된다. 초보자는 이 오류를 "$ERROR 블록 문제"로 오진하여 수일을 소비한다.

### C. Professional Moat

> "NONMEM을 실행하고 파라미터를 추출하는 것은 이미 자동화 가능하다. 이 섹션의 코드 구조와 알고리즘 수학을 진정으로 내면화한 모델러만이 — 수렴 실패, \$COV step 에러, CWRES 편향, 예상치 못한 OFV 상승 앞에서 — 무엇을 할 수 있는가?"

**정밀한 답:** 데이터셋 구조와 예약 변수 체계를 내면화한 모델러는 **진단의 출발점을 모델 코드가 아닌 데이터셋으로 설정**할 수 있다. 구체적으로:

- CWRES의 시간축 편향 클러스터를 보고 "이것이 모델 구조 문제인가, ADDL 오기인가, SS 코딩 오류인가"를 **데이터셋 레벨에서 먼저 분리**한다. AI는 OFV를 최소화하는 파라미터 조합을 찾을 수 있지만, 데이터셋의 TIME 오기 또는 SS 오코딩이 만들어내는 "잘못된 패턴에 완벽히 맞는 파라미터"를 탐지할 수 없다.
- 정주 투여 + 경구 유지요법 혼합 데이터에서 ADVAN2·CMT·EVID를 조합하여 단일 데이터셋에서 두 투여 경로를 동시에 처리하는 구조를 설계한다.
- 규제 제출용 데이터 명세서(dataset specification document)에서 각 예약 변수의 사용 근거와 QC 검증 이력을 FDA/EMA 심사관이 재현 가능한 언어로 기술한다.

이것은 모델 선택이나 추정 방법의 문제가 아니라 **데이터와 모델이 만나는 경계면(interface)을 설계하는 능력**이다 — 그리고 이 능력은 예약 변수 체계의 구조적 필연을 이해하지 못하면 획득되지 않는다.
