NONMEM 데이터셋

Owen & Fiedler-Kelly, Ch.4 pp.66–89  |  PIPET 교재 Ch.2 pp.13–25

Big Idea

NONMEM 데이터셋은 단순한 스프레드시트가 아니라 "약물이 몸에 들어가고 측정되는 시간적 사건의 시퀀스를 기계가 읽을 수 있는 유일한 언어로 번역한 것"이며, 이 번역의 오류는 모델 추정치를 조용히, 그러나 치명적으로 오염시킨다.

개념 해부 카드

C-1 데이터셋 구조 원칙
개념 Big Idea: 거장의 시각

왜 치명적으로 중요한가: 투약 레코드의 TIME이 관측 레코드보다 1초라도 뒤에 놓이거나, 동일 레코드에 AMT와 DV가 함께 존재하면 — NONMEM은 에러를 내거나 더 나쁜 경우, 조용히 잘못된 PK 파라미터를 추정한다. NDA 제출 직전 데이터셋 오류가 발견된 사례는 드물지 않다.

체화해야 할 단 하나의 직관: 데이터셋은 "행 = 한 순간에 일어난 단 하나의 사건"이라는 원칙 위에 서 있다. 투약과 관측은 절대 같은 행에 공존할 수 없다 — 현실에서 동시에 일어나더라도 두 행으로 분리한다.

이 직관을 머릿속에 박고 아래를 읽어라: 모든 규칙이 이 한 가지 원칙의 파생임을 기억하라.

A. 정의

NONMEM(NM-TRAN) 분석 데이터셋은 ASCII 형식 평면 파일(flat file)로, 행(레코드)과 열(데이터 항목)의 2차원 배열이다. 각 레코드는 특정 시점에 동시적으로 발생한 사건의 집합이다. PREDPP를 사용하는 모든 모델은 데이터를 시간순 이벤트(time-ordered events)로 처리하며, 레코드는 세 유형으로 분류된다: 투약 이벤트(Dosing event), 관측 이벤트(Observation event), 기타 유형 이벤트(Other-type event).

B. 코드 구조

동일 시점에 투약과 관측이 발생하는 경우 — 반드시 두 행으로 분리:

; ✅ 올바른 코딩 — 동일 시점 이벤트의 두 행 분리
ID  TIME  AMT    DV     MDV
001  0.0   100    .      1    ← 투약 레코드: DV 결측
001  0.0   .      35.4   0    ← 관측 레코드: AMT 결측
001  0.5   .      12.6   0

; ❌ 잘못된 코딩 — AMT + DV 동일 레코드 (NM-TRAN 에러 또는 무결성 훼손)
ID  TIME  AMT    DV
001  0.0   100    35.4    ← NONMEM이 거부하거나 예측 불능 상태
[출처: Owen Ch.4, Table 4.3–4.4, p.70]
D. 가정 & 경계조건
  • 모든 레코드는 동일한 열 수를 유지. 누락값은 "." 또는 "0"으로 채운다. 빈칸은 허용되지 않는다(space-delimited 파일에서).
  • 각 개체의 레코드는 연속적으로(contiguous) 배열. ID가 섞이면 NONMEM은 분리된 ID를 다른 개체로 처리한다.
  • TIME은 개체 내에서 단조 증가(non-decreasing). 단, 재설정 이벤트(EVID=3/4)가 있을 경우 예외.
  • NONMEM 7 기준 한 번의 실행에서 처리 가능한 최대 데이터 항목 수: 50개. 초과 시 DROP 옵션 활용 필수.
G. Zettelkasten Atom

NONMEM PREDPP 데이터셋은 시간순 이벤트 스트림이다. 투약 레코드(AMT ≠ 0, DV = missing)와 관측 레코드(DV ≠ missing, AMT = missing)는 절대 동일 행에 공존할 수 없다 — ODE 적분기의 분기 논리가 이를 요구하기 때문이다. 개체 레코드는 연속 배열(contiguous)되어야 하며, TIME은 개체 내 단조 증가해야 한다.

C-2 예약 변수 체계 ⚡ Apex Concept
개념 Big Idea: 거장의 시각

왜 치명적으로 중요한가: NONMEM에 내장된 예약 변수명을 잘못 이해한 채 데이터셋을 만들면, NONMEM은 에러 없이 실행되지만 완전히 다른 모델을 추정한다 — EVID의 오기는 특정 관측값을 피팅에서 제외하거나, SS=1 누락은 수백 개의 허구 투약 이벤트를 삽입시킨다.

체화해야 할 단 하나의 직관: 예약 변수는 NONMEM과 사용자 사이의 "계약서 조항"이다. 각 변수의 값은 "지금 이 행에서 시스템 상태가 어떻게 변하는지"를 NONMEM에게 전달하는 신호이지, 단순한 숫자가 아니다.

이 직관을 머릿속에 박고 아래를 읽어라: 아래 각 변수를 "숫자"가 아닌 "시스템 상태 신호"로 읽으면 규칙이 전부 자연스럽게 도출된다.

A. 예약 변수 전체 목록
변수상황의미
ID항상개체 식별자 (숫자, 연속 배열 권장)
DV항상종속변수 — 피팅 대상 관측값
MDV항상결측 종속변수 플래그 (1=결측, 0=존재)
TIMEPREDPP이벤트 시간 (단조 증가, non-negative)
AMTPREDPP투여 용량 (투약 레코드에만, 관측 레코드는 결측)
RATEPREDPP투여속도: 양수=직접 지정, 0=볼루스, −1=속도 추정, −2=기간 추정
ADDLPREDPP추가 투여 횟수 (첫 투여 제외한 N−1)
IIPREDPP투여 간격 (ADDL/SS와 함께 사용)
SSPREDPP항정상태 플래그 (0/1/2)
CMTPREDPP구획 지정 (투약·관측 구획 분리 시 필수)
EVIDPREDPP이벤트 유형 명시적 선언 (0/1/2/3/4)
[출처: PIPET Ch.2, 표 2.2, p.19 + Owen Ch.4, §4.3, pp.71–80]
B. EVID 체계 완전 해부
EVID=0  → 관측 이벤트     (DV 필요, AMT 결측 필수)
EVID=1  → 투약 이벤트     (AMT 필요, DV 결측 필수)
EVID=2  → 기타 유형 이벤트 (공변량 갱신, 예측값 반환용)
EVID=3  → 시스템 리셋        (구획 내 양을 0으로 초기화)
EVID=4  → 리셋 + 즉시 투약   (EVID=3 + EVID=1 조합)

4     Required to be missing Required to be nonmissing

C-2. Plausible Fallacy (그럴싸한 오류)
⚠ 가장 자주 저지르는 오류

오류: "EVID는 어차피 MDV로 대체 가능하다 — EVID=2는 그냥 MDV=1로 처리하면 된다."

나비효과 1 — 교차시험 세션 간 농도 오염: EVID=3(리셋)을 사용하지 않으면 Period 1의 잔류 약물 양이 Period 2 적분에 포함된다. 결과: Period 2의 CL이 체계적으로 과소추정되고, 이 편향이 IIV 추정치로 흡수되어 η-shrinkage가 비정상적으로 증가한다.

나비효과 2 — PK/PD 링크 단절: EVID=2 레코드를 MDV=1로 대체하면 해당 시점의 예측값이 $TABLE에 반환되지 않는다. PK/PD 링크 모델에서 PD 구획에 공급되는 농도(Cpred)가 비어 모델 전체가 붕괴된다.

GOF 시그니처 — "Cross-Period Phantom Elevation": CWRES vs. TIME 플롯에서 각 Period 초반(0–2×t½ 구간)에 양의 방향 편향 클러스터가 반복적으로 나타난다. "흡수 모델 문제"로 오해하기 쉽지만 실제 원인은 이전 Period 잔류 약물의 잘못된 중첩이다.

G. Zettelkasten Atom

NONMEM 예약 변수는 사용자-시스템 간 계약 조항이다. EVID(0/1/2/3/4)는 각 레코드의 이벤트 유형을 명시하며, MDV는 DV를 피팅에 포함할지 여부를 제어한다. 두 변수는 독립적이지만 상호 보완적이다. EVID=2는 MDV=1이면서도 모델 예측값을 반환시키는 고유 기능을 가진다 — 이를 MDV=1로 대체하면 PK/PD 링크 모델이 붕괴된다.

C-3 투여 모델링 고급 기법 — ADDL/II·SS·RATE
개념 Big Idea: 거장의 시각

왜 치명적으로 중요한가: Phase 3 임상 데이터에서 환자가 수개월간 하루 두 번 복용했다면, ADDL에 총 투여 횟수(N)를 입력하면 약 하나가 더 투여된다 — 이 오류는 CL을 체계적으로 과소추정시킨다.

체화해야 할 단 하나의 직관: ADDL은 "추가로 더 투여한 횟수"이지 "총 투여 횟수"가 아니다 — ADDL = N−1.

이 직관을 머릿속에 박고 아래를 읽어라: SS·RATE의 값 체계도 "첫 이벤트는 레코드에 명시, 나머지는 묵시적 선언"이라는 같은 원칙으로 작동한다.

B. ADDL + II 예시
; 경구 항생제 250mg, 하루 2회 10일간 → 총 20회 투여 (Owen Table 4.15)
ID  TIME  AMT  ADDL  II   DV    EVID  MDV  CMT  TAD
1   0     250  19    12   .     1     1    1    0
1   1.0   .    .     .    10.6  0     0    2    1.0
1   220   .    .     .    12.3  0     0    2    4.0

; ADDL=19: TIME=0에 첫 투약(명시) + 12시간 간격 19회 추가(묵시) = 총 20회
; ADDL = 20 - 1 = 19  ← 반드시 N-1
[출처: Owen Ch.4, Table 4.15, p.82]

SS 값 체계:

SS=0  비항정상태 — 이전 투약 이력 그대로 유지
SS=1  항정상태 → 구획 내 양을 이론적 SS 조건으로 재설정
      이전 투약 레코드 효과는 무시(reset). II 반드시 명시.
SS=2  항정상태 + 이전 투약 이력 유지 (superposition)
      → 예정에 없던 추가 투여가 있었을 때 사용

RATE 값 체계:

RATE = 양수   직접 투여 속도 지정 (AMT per unit TIME). 지속시간 = AMT/RATE
RATE = 0      볼루스 투여 (정주입이 아님)
RATE = -1     투여속도를 파라미터로 추정 → $PK에 R1=THETA(n) 필요
RATE = -2     투여지속시간을 파라미터로 추정 → $PK에 D1=THETA(n) 필요
[출처: Owen Ch.4 §4.3.9–4.3.12 + PIPET §2.5.7–2.5.9, pp.77–80]
B-2. SS=2 적용 예시 — 만성 투여 중 비정기 추가 투여
; 매일 7:00AM 250mg 만성 투여 + 11/20 18:00 추가 투여 → 11/22 10:30 채혈
ID  DATE      TIME   AMT  SS  II  DV    EVID  MDV  CMT  TAD
1   11/20/10  18:00  250  0   .   .     1     1    1    0   ← 추가 투여 (비항정상태)
1   11/22/10  7:00   250  2   24  .     1     1    1    0   ← 만성 SS 투여 (superposition)
1   11/22/10  10:30  .    .   .   23.4  0     0    2    3.5
[출처: Owen Ch.4, Table 4.18, p.84]
C-4 제어구문 연결 — $PROB · $DATA · $INPUT
개념 Big Idea: 거장의 시각

왜 치명적으로 중요한가: $INPUT의 열 선언이 데이터셋 실제 열 순서와 어긋나면 — NONMEM은 에러 없이 실행하지만 잘못된 열을 AMT로 읽는다. 이 오류는 출력 파일에서는 거의 보이지 않는다.

체화해야 할 단 하나의 직관: $INPUT은 데이터셋 열의 "이름표(label)를 순서대로 붙이는 선언"이다 — 데이터셋 자체는 바뀌지 않는다.

이 직관을 머릿속에 박고 아래를 읽어라: $DATA의 IGNORE/ACCEPT는 데이터셋을 필터링하고, $INPUT은 열에 이름을 부여하며, $PROB는 실행 이력을 관리한다 — 세 레코드의 역할은 완전히 분리되어 있다.

B. 제어구문 필수 3행
$PROBLEM HO_IV_1comp_BaseModel_v1
; ↑ 버전 관리 핵심 — 모델 복잡도 증가 시 v2, v3로 변경

$DATA ../HOdata_IV.csv IGNORE=@
; ↑ IGNORE=@: 첫 글자가 알파벳인 행 제거 = 헤더 제거
; IGNORE=(SEX.EQ.1): 조건부 데이터 필터링

$INPUT ID TIME AMT DV MDV SEX AGE WT HT
; ↑ 데이터셋 열 순서와 반드시 1:1 일치
; 불필요 열: RACE=DROP
; 변수명 재매핑: AMT=DOSE (데이터셋의 DOSE 열을 AMT로 인식)
[출처: PIPET Ch.2, §2.1, pp.14–15 + Owen Ch.4, §4.2, p.68–69]
B-2. 단위 통일 전략 (AMT-DV 단위 불일치 해결)
; 방법 1: 데이터셋에서 미리 변환
; 투여 용량 mg → ng: AMT = 100,000 (ng), DV = 891.2 (ng/mL)
; PIPET 표 2.1의 실제 예시 수치

; 방법 2: $PK에서 scaling factor 적용
$PK
  S1 = V/1000  ; AMT(mg) / V(L) = mg/L, ×1000 = µg/mL
               ; S1 = V → C = A1/S1 통해 mg/L 단위 달성
[출처: PIPET Ch.2, §2.5.6, p.22]
C-5 공변량·시변 데이터·헤더 처리
개념 Big Idea: 거장의 시각

왜 치명적으로 중요한가: 장기 연구에서 체중이 변화할 때 ADDL로 지정된 묵시적 투약 레코드의 공변량 값은 이전 레코드가 아닌 다음 물리적 레코드의 값을 사용한다 — 이를 모르면 체중 기반 공변량 모델이 의도와 다른 시점에서 체중을 바꾸게 된다.

체화해야 할 단 하나의 직관: NONMEM은 묵시적 이벤트의 공변량을 "미래로부터" 가져온다.

이 직관을 머릿속에 박고 아래를 읽어라: $BIND 구문은 이 기본값을 제어하는 고급 기법이다.

§3

구조적 동형성 지도

공통 원리: 상태-이벤트 로그(State-Event Log)의 보편성

NONMEM 데이터셋의 핵심 구조 — "이벤트가 시스템 상태를 변경하거나 측정하며, 그 순서가 보존된다" — 는 자연계와 정보시스템 전반에서 동일하게 나타난다. 이 구조는 중첩 원리(superposition principle)를 수학적 기반으로 한다: 복수의 독립적 사건 효과는 선형적으로 합산된다.

왜 이 형태인가: 연속 시간계에서 임의의 입력이 가해지는 선형 시스템은 각 입력의 임펄스 응답(impulse response)의 합으로 완전히 표현된다. NONMEM 데이터셋은 이 수학적 필연을 "레코드 = 사건, 순서 = 시간축"으로 구현한 것이다.

예측 전이: 이 동형성을 이해하면, 데이터베이스 트랜잭션 로그나 EHR 투약 기록의 설계 원칙을 NONMEM 데이터셋 QC에 직접 전용할 수 있다.

Layer 4 — 예측 전이 카드

🔁 전이 질문 1: 데이터베이스 설계자라면 NONMEM 데이터셋의 "투약과 관측이 같은 레코드에 있으면 안 된다"는 규칙을 보고 어떤 설계 원칙을 즉각 인식하는가? ▼
① 출발 도메인 — 관계형 DB 설계: 정규화(Normalization) 1NF 원칙은 "하나의 레코드는 하나의 원자적 사실만 표현해야 한다." 투약 사건과 관측 사건은 서로 다른 원자적 사실이다.
② 공통 수학 구조 — 사건 분리의 필연: 시스템 상태를 추적하는 모든 이벤트 로그는 "입력(input) 이벤트"와 "측정(measurement) 이벤트"를 분리해야 한다. 이 둘이 동일 레코드에 있으면 시스템은 어느 것을 처리해야 할지 분기할 수 없다.
③ 대응 현상 도출: 관계형 DB의 1NF, NONMEM의 "AMT/DV 동시 금지", HL7 FHIR의 "Medication + Observation 별도 리소스" — 모두 같은 원칙의 도메인별 구현이다. DB 설계자는 NONMEM 규칙을 처음 보는 순간 "아, 1NF 원칙이구나"라고 즉시 인식할 수 있다.
🔁 전이 질문 2: Git에서 "commit은 되돌릴 수 없고 시간 순서가 보존된다"는 원칙을 알면, NONMEM 데이터셋에서 왜 TIME이 개체 내에서 단조 증가해야 하는지 즉시 예측할 수 있는가? ▼
① 출발 도메인 — Git 커밋 로그: Git은 각 커밋을 SHA-1 해시로 연결하며 시간 순서를 보존한다. 과거 커밋을 현재 커밋 사이에 삽입하는 것은 불가능하다 — 이미 발생한 사건은 순서가 고정된다.
② 공통 수학 구조 — 인과율(Causality): 이벤트 기반 시스템에서 t₂ < t₁인 이벤트가 t₁ 이후에 처리되면 인과율이 위반된다. ODE 적분기는 시간 전진 방향으로만 적분할 수 있다.
③ 대응 현상 도출: Git의 "시간순 DAG", NONMEM의 "TIME 단조 증가 필수", 금융 원장의 "거래 타임스탬프 불가역성" — 동일 원칙이다. Git에서 rebase가 히스토리를 재작성하는 "특수 작업"이듯, NONMEM에서 EVID=3(리셋)도 특수 선언이 필요한 예외적 작업이다.

혼동쌍 해부

혼동쌍 1 — MDV vs. EVID

비교 차원MDVEVID
표면적 유사성둘 다 "이 레코드가 관측값을 포함하는가"를 묻는 것처럼 보인다
코드 형태MDV = 0/1EVID = 0/1/2/3/4
구조적 지시체"이 행의 DV를 OFV 계산에 포함시킬 것인가?" — 피팅 포함 여부"이 행에서 시스템에 어떤 사건이 일어나는가?" — 사건 유형 선언
핵심 차이MDV=1이어도 EVID=2이면 모델 예측값이 반환됨EVID=2는 MDV=1을 내포하지만, 역은 성립 안 함
임상/모델링 결과MDV만 사용하고 EVID 생략 시: 교차시험 period 간 농도 중첩, PK/PD 링크 모델 예측값 누락
⚡ 기억 고리MDV = 채점표 (이 관측값 점수를 줄 것인가?) · EVID = 출석부 (오늘 어떤 수업이 있었는가?) — 채점표에 0이어도 수업 종류는 별도로 선언해야 한다.

혼동쌍 2 — SS=1 vs. SS=2  Critical Blow

비교 차원SS=1SS=2
표면적 유사성둘 다 "항정상태 투약"을 선언한다
코드 형태SS=1, II=τSS=2, II=τ
이전 투약 이력모두 무시(reset) → 이론적 SS 구획 양으로 재설정이전 이력 유지(superposition) → SS 조건 중첩
적용 상황교차시험 Period 2 첫 투약 — 세척 기간 후 새로 시작만성 투여 중 비정기 추가 투여 — 기존 SS에 중첩
혼동 시 결과SS=1이어야 할 곳에 SS=2 사용 시: Period 1 잔류 약물이 유령처럼 중첩 → CL 과소추정, η-shrinkage 팽창
⚡ 기억 고리SS=1은 "백지에서 다시 시작"(reset + SS 계산) · SS=2는 "이전 장에 새 이야기를 덧쓰기"(superposition) — 교차시험이면 백지, 만성 중 추가 투여면 덧쓰기
치명적 타격: 교차시험 데이터에서 SS=1 대신 SS=2를 사용해 NDA 제출하면: 심사관이 Period별 PK 파라미터 비교에서 체계적 편향(systematic bias)을 발견하여 Deficiency Letter를 발행하고, 재분석을 요구한다. 이미 제출된 용량 권고도 재검토 대상이 된다.

혼동쌍 3 — ADDL vs. N (총 투여 횟수)

비교 차원ADDLN (총 횟수)
표면적 유사성둘 다 "몇 번 투여했는가"를 나타내는 것처럼 보인다
정의ADDL = N − 1 (레코드에 명시된 첫 투여를 제외한 추가 횟수)N = 총 투여 횟수 (레코드의 첫 투여 포함)
실수 시나리오N=10 투여에서 ADDL=10 입력 → 실제로 11회 투여가 시뮬레이션됨
임상/모델링 결과총 AUC 약 10% 과대추정, CL 과소추정 — 소규모 데이터셋에서 θ(CL) 편향이 η로 흡수
⚡ 기억 고리"ADDL은 '추가'다 — 이미 레코드에 한 번이 있으니 거기서 하나 뺀다." ADDL = N−1은 수학 공식이 아니라 언어 정의다: Additional = 이미 있는 것에 더하는 것

SIGNATURE-1: "Cross-Period Phantom Elevation"

GOF 플롯 패턴: CWRES vs. TIME에서 교차시험 각 Period 초반 0–2×t½ 구간에 양의 방향 편향 클러스터가 반복 출현. NPDE 분포가 기대값보다 음의 방향으로 치우침.

기계론적 원인: SS=1이어야 할 레코드에 SS=2 코딩 → NONMEM이 이전 Period 잔류 약물을 현재 Period 초기 구획 양에 중첩. 모델은 실제보다 높은 초기 농도를 예측하므로 관측값이 상대적으로 낮아 보임 → 양의 CWRES.

해결 방향: SS 코딩 전면 재검토. 교차시험 Period 시작 레코드를 모두 SS=1로 수정. 충분한 세척 기간(≥5×t½) 미충족 시에는 Period 1 마지막 농도를 초기조건($PK의 A_0)으로 직접 지정.

η-shrinkage 관련: 이 오류는 CL η-shrinkage를 비정상 팽창시킨다. η-shrinkage > 30%와 상기 CWRES 패턴이 동시에 관찰되면 SS 코딩 오류를 최우선 의심해야 한다.

SIGNATURE-2: "Phantom Dose Overdose" — ADDL=N 오기

GOF 플롯 패턴: DV vs. PRED 플롯에서 전체 데이터에 걸쳐 경미하지만 일관된 과소추정(PRED > DV). OFV는 정상 범위이지만 CL 추정치가 문헌 기반 예상값보다 10–20% 낮음.

기계론적 원인: ADDL=N(총 횟수)으로 오기 → NONMEM이 N+1회 투여를 시뮬레이션. 실제보다 한 번 더 투여가 있었다고 가정하므로 농도를 과대추정하고, 이를 보정하기 위해 CL을 과소추정한다.

해결 방향: 데이터셋의 모든 ADDL 값을 N−1로 수정. QC 단계에서 ADDL×II 구간의 마지막 묵시적 투약 시각이 다음 관측 레코드 TIME보다 반드시 앞에 있는지 검증하는 로직 추가.

ε-shrinkage 관련: 잔차 에러 구조가 정상임에도 ε-shrinkage가 높고 η-shrinkage가 낮은 경우, 데이터셋 수준의 투여 이력 오류를 의심한다 — 개체 파라미터가 아닌 모집단 파라미터가 편향되어 있기 때문이다.

BOSS DILEMMA Q9 — Phase 3 다기관 연구 최종 분석 직전 QC에서 발견: 전체 1,247명 중 23명의 교차시험 Period 2 첫 투약 레코드에 SS=2가 기재됨(SS=1이어야 했던 레코드). 수정 시 데이터셋 버전 변경과 재분석이 필요하며, 규제 제출 일정이 3주 지연된다. 민감도 분석에서 이 23명(1.8%)을 제외해도 최종 CL 추정치는 4.2% 차이. 당신은 수정 후 재분석(일정 3주 지연)을 택하는가, 아니면 민감도 분석 결과를 첨부하고 현재 데이터셋으로 제출하는가? SR ★★ ▼

수석 모델러의 답변: 반드시 수정 후 재분석한다.

Trade-off 분석 — 제출 강행의 유혹: 23명/1247명(1.8%)은 통계적으로 소수다. 4.2% 차이는 CL 추정의 불확실도 범위 내일 수 있다. 3주 지연은 상업적 비용이 크다.

그러나 반드시 수정해야 하는 세 가지 구조적 이유:

1. 데이터셋 오류는 "알려진 편향(known bias)"이다. 불확실성(uncertainty)과 편향(bias)은 근본적으로 다르다. 심사관은 랜덤 변동성을 허용하지만 알려진 체계적 오류는 허용하지 않는다. 심사관이 독립적으로 재현 분석했을 때 이 오류를 발견하면 Deficiency Letter가 아닌 Complete Response Letter로 이어질 수 있다.

2. ICH E9(R1)과 FDA Population PK 가이던스는 데이터 무결성을 요구한다. SS 오코딩은 모델이 아닌 데이터 레벨의 오류다. 민감도 분석은 모델 가정의 변동을 탐색하는 도구이지, 알려진 데이터 오류를 정당화하는 도구가 아니다.

3. 규제 방어 가능성: 수정 후 재분석 보고서에서는 "오류 발견 → 수정 → 재분석 → 결론 안정성 확인"의 완결된 QC 이력을 제시할 수 있다.

"A dataset error was identified during final QC review: 23 subjects in the crossover cohort had SS=2 coded for the first dose of Period 2, where SS=1 was indicated. The dataset was corrected, and analysis was repeated. The final PK parameter estimates differed by less than 5% from the pre-correction analysis, confirming the robustness of the model."

결론: 3주 지연의 단기 비용보다 데이터 무결성 훼손의 장기 규제 리스크가 훨씬 크다.

C. Professional Moat

"NONMEM을 실행하고 파라미터를 추출하는 것은 이미 자동화 가능하다. 이 섹션의 코드 구조와 알고리즘 수학을 진정으로 내면화한 모델러만이 — 수렴 실패, $COV step 에러, CWRES 편향, 예상치 못한 OFV 상승 앞에서 — 무엇을 할 수 있는가?"

정밀한 답: 데이터셋 구조와 예약 변수 체계를 내면화한 모델러는 진단의 출발점을 모델 코드가 아닌 데이터셋으로 설정할 수 있다. CWRES의 시간축 편향 클러스터를 보고 "이것이 모델 구조 문제인가, ADDL 오기인가, SS 코딩 오류인가"를 데이터셋 레벨에서 먼저 분리한다.

AI는 OFV를 최소화하는 파라미터 조합을 찾을 수 있지만, 데이터셋의 TIME 오기 또는 SS 오코딩이 만들어내는 "잘못된 패턴에 완벽히 맞는 파라미터"를 탐지할 수 없다.

정주 투여 + 경구 유지요법 혼합 데이터에서 ADVAN2·CMT·EVID를 조합하여 단일 데이터셋에서 두 투여 경로를 동시에 처리하는 구조를 설계한다. 규제 제출용 데이터 명세서(dataset specification document)에서 각 예약 변수의 사용 근거와 QC 검증 이력을 FDA/EMA 심사관이 재현 가능한 언어로 기술한다.

이것은 모델 선택이나 추정 방법의 문제가 아니라 데이터와 모델이 만나는 경계면(interface)을 설계하는 능력이다 — 그리고 이 능력은 예약 변수 체계의 구조적 필연을 이해하지 못하면 획득되지 않는다.
