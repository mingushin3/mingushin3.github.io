Pharmacometrics · LLM-Driven Preprocessing Protocol

NONMEM-Ready 55-Step Sequential Question Protocol

어떤 데이터셋이든 Q01부터 순서대로 LLM에게 물어가면, 중간에 조기 완료되거나 Q55에서 최종 완료되어 NONMEM-ready 데이터셋이 만들어지는 마법의 질문 경로.
깨끗한 SDTM은 ~Q35에서, 비표준 Excel은 Q55까지 — 데이터 상태에 따라 경로가 달라지되 도착지는 동일하다.

총 질문 55개
필수(MUST) 32개
조건부(COND) 15개
자동화(AUTO) 8개
Phase 8개
사용법

Step 1. 데이터 파일을 LLM에게 업로드한다.

Step 2. 아래 질문을 Q01부터 순서대로 복사하여 LLM에게 던진다. 각 질문의 프롬프트 박스를 그대로 사용하거나, 자신의 데이터에 맞게 변수명만 바꿔 사용한다.

Step 3. LLM이 R 코드로 처리 결과를 돌려주면 확인 후 다음 질문으로 넘어간다. COND 뱃지가 붙은 질문은 해당 조건이 아니면 건너뛴다.

Step 4. Q50–Q55의 최종 QC를 통과하면 NONMEM-ready 완성.

💡 깔끔한 SDTM 데이터 → ~Q35에서 조기 완료 가능 · 비표준 Excel → Q55까지 풀코스 · 기존 NONMEM 수정 → Q20대에서 완료 가능

Q01–Q05
Q01 데이터 형태 식별 & 파일 인벤토리
MUST
업로드한 데이터 파일들을 확인하고, (1) 파일 형식 (xpt/csv/xlsx/sas7bdat), (2) 데이터 유형 (SDTM/ADaM/Raw CRF/Excel/기존 NONMEM), (3) 각 파일의 행·열 수와 변수 목록을 표로 정리해줘. 파일이 여러 개면 각각 정리하고, SDTM이면 도메인명(DM, EX, PC 등)도 식별해줘.
LLM이 확인할 것
파일 확장자로 형식 판별 → haven::read_xpt(), readr::read_csv(), readxl::read_excel() 등 적절한 reader 선택. SDTM이면 DOMAIN 변수 또는 파일명에서 도메인 식별. ADaM이면 ADSL/ADPC/ADEX 존재 확인. 기존 NONMEM이면 EVID, MDV, AMT 컬럼 존재로 판별.
→ Phase 0: 전체 소스 유형 분기
Q02 Subject ID 식별 & 고유 대상자 수 확인
MUST
데이터에서 대상자 식별 변수(USUBJID, SUBJID, PTNO 등)를 찾고, 고유 대상자 수를 세어줘. ID 형식이 문자열이면("S-001", "SITE01-003" 등) 숫자 ID로 재코딩하는 매핑 테이블도 만들어줘. 여러 파일에서 ID 교집합/차집합도 확인해줘.
LLM이 확인할 것
ID 중복 없는지, 파일 간 ID 일치하는지, 숫자 변환 시 충돌 없는지. SDTM이면 USUBJID가 표준. multi-study면 STUDYID × SUBJID 조합으로 unique ID 생성.
id_map <- dm %>% distinct(USUBJID) %>% mutate(ID = row_number())
→ Phase 0: Subject ID 체계 통일
Q03 날짜/시간 변수 식별 & 형식 평가
MUST
모든 파일에서 날짜/시간 관련 변수를 찾고, 각각의 형식(ISO 8601? AM/PM? 날짜만? 날짜+시간?), 결측률, 파싱 가능 여부를 표로 정리해줘. 특히 투여 시점(EXSTDTC 등)과 채혈 시점(PCDTC 등)의 시간 정밀도(분 단위까지 있는지)를 확인해줘.
왜 중요한가
TIME 변수 계산의 원재료. 시간 정밀도가 PK 분석 정밀도를 결정. PD-only면 날짜만으로 충분할 수 있음. 날짜/시간이 분리된 컬럼인 경우 합쳐야 함. AM/PM, locale 혼재 시 변환 로직 필요.
→ Phase 2 선행 조건
Q04 핵심 소스 테이블 매핑
MUST
이 데이터에서 NONMEM building block 3개의 소스를 매핑해줘:
• Dosing 소스: 어떤 파일/테이블의 어떤 변수가 투여 기록인가? (AMT, 투여일시, 투여경로)
• Observation 소스: PK 농도(DV) 데이터는 어디에? 단위는? BLQ 표시는?
• Covariate 소스: demographics, lab, vitals는 어디에?
각각에 대해 [소스파일 → 핵심변수 → NONMEM 변수] 매핑 테이블을 만들어줘.
LLM이 확인할 것
SDTM이면: EX→dosing, PC→observations, DM+LB+VS→covariates.
ADaM이면: ADEX→dosing(재구성 필요), ADPC→observations, ADSL→covariates.
Raw CRF/Excel이면: DDT(Data Definition Table) 작성이 사실상 이 단계.
→ Phase 0: DDT 수립 / 6대 표준 입력 확인
Q05 기존 NONMEM 데이터셋 인수인계 체크
COND기존NM만
이 데이터셋은 기존 NONMEM 데이터셋이다. (1) TIME 기준점(reference time)이 무엇인지 역추적해줘. (2) FLAG/IGNORE/EXCL 컬럼 convention을 파악해줘. (3) $INPUT에 나열된 변수 순서를 데이터 컬럼 순서와 대조해줘. (4) 새로 추가할 데이터가 있다면 ID 충돌 방지 방안도 제안해줘.
⏭ 기존 NONMEM 데이터를 수정/확장하는 경우에만 해당. 새로 만드는 경우 건너뛰기.
→ Phase 0: 기존 데이터셋 수정/확장
1a
Dosing Records 조립
EVID=1 — 전체 작업의 가장 중요한 블록
Q06–Q15
Q06 투여 기록 추출 & 기본 구조 생성
MUST
dosing 소스에서 투여 기록을 추출해서 다음 컬럼을 가진 dosing 데이터프레임을 만들어줘: ID, DATETIME, AMT, CMT, EVID=1, MDV=1, DV=0. 각 ID별 투여 횟수, 총 투여 record 수를 요약해줘.
핵심 체크
EX 도메인이면 EXSTDTC(투여 시작), EXDOSE(용량) 사용. 실제 투여 여부 확인: EXSTAT이 "NOT DONE"인 건 제외. 모든 dosing record에 EVID=1, MDV=1 설정.
→ Phase 1: Dosing records 필수 변수
Q07 Actual vs Planned dose 확인
MUST
AMT에 들어간 값이 실제 투여량(actual dose)인지 계획 투여량(planned dose)인지 확인해줘. Actual과 planned가 다른 record를 찾아 목록으로 보여줘. AMT는 반드시 actual dose여야 하고, planned dose는 별도 PDOSE 컬럼으로 보존해줘.
부분 주입, 구토, dose reduction 등으로 actual ≠ planned인 경우 존재. EX 도메인: EXDOSE가 actual이 일반적이나, EXADJ 등 조정 컬럼 확인.
→ Phase 1: Actual vs Planned dose
Q08 투여 경로 & RATE/DUR 설정
MUST
투여 경로(PO/IV bolus/IV infusion/SC/IM)를 확인하고, 경로에 맞게 RATE와 DUR 컬럼을 설정해줘:
• 경구/SC/IM bolus → RATE=0 (또는 생략)
• IV bolus → RATE=0
• IV infusion → RATE=AMT/주입시간(hr) 또는 RATE=-2 + DUR=주입시간(hr)
주입 시간 정보가 어디 있는지도 찾아줘.
EX 도메인: EXROUTE(경로), EXSTDTC~EXENDTC 차이로 infusion duration 계산 가능. 혼합 경로(예: loading IV → maintenance PO)이면 각각 다른 CMT 할당 고려.
→ Phase 1: 투여 경로별 의사결정 트리
Q09 AMT 단위 확정 & 변환
MUST
AMT의 단위를 확인하고 확정해줘(mg? μg? cells?). DV 단위와의 조합이 PK 파라미터 단위를 결정하므로, AMT(단위) ÷ DV(단위) → CL(단위) 관계를 명시해줘. 단위 변환이 필요하면 수행하고 변환 로직을 기록해줘.
가장 흔한 조합: AMT(mg), DV(ng/mL=μg/L) → CL(L/hr), V(L).
체중 기반 투여(mg/kg)면 AMT는 absolute dose(mg)로 변환. 체중은 WT 컬럼으로 분리.
→ Phase 1: 단위 / Phase 3: 체중 기반 투여
Q10 CMT 할당 (Dosing compartment)
MUST
dosing record의 CMT(compartment) 번호를 할당해줘. 일반적으로: 경구 투여 → CMT=1 (depot/absorption compartment), IV → CMT=2 (central). 투여 경로가 혼합이면 경로별 CMT 매핑을 명시해줘.
CMT 번호는 NONMEM control stream의 $MODEL 정의와 반드시 일치해야 함. 여기서는 데이터 수준에서 잠정 할당하고, 최종은 모델러와 협의.
→ Phase 1: CMT 설정
Q11 반복 투여 압축 (ADDL/II)
COND반복투여
동일 용량의 반복 투여 구간을 찾아서 ADDL(추가 투여 횟수)과 II(투여 간격, hr) 컬럼으로 압축해줘. 단, 용량 변경 시점에서는 ADDL을 끊고 새 dosing record를 시작해줘. 압축 전후 총 투여 횟수가 동일한지 검증해줘.
예: 매일 100mg × 28일 → 1행: AMT=100, ADDL=27, II=24.
Day 15에 150mg으로 변경 → 2행: Day1 AMT=100, ADDL=13, II=24 + Day15 AMT=150, ADDL=13, II=24.
⏭ Single dose study이면 건너뛰기. ADDL/II 없이 각 투여를 개별 행으로 유지해도 무방.
→ Phase 1: 반복 투여 패턴 / ADDL/II
Q12 투여 중단/재시작 & Reset (EVID=4)
COND
투여 중단 후 재시작이 있는 대상자가 있는지 확인해줘. Washout 후 재투여, 2nd course 시작 등이면 EVID=4(reset+dose)가 필요한지 판단해줘. 중단 기간을 식별하고, 재시작 시 reset 적용 여부를 결정해줘.
EVID=4: 모든 compartment를 0으로 초기화한 뒤 투여. 장기 washout 후 재투여에 적합.
EVID=3: reset만 (투여 없음). 드물게 사용.
⏭ 연속 투여 프로토콜이고 중단/재시작이 없으면 건너뛰기.
→ Phase 1: Reset (EVID=4)
Q13 Steady-state 처리 (SS/II)
COND
이전 투여 이력이 불완전한 대상자가 있는지 확인해줘 (예: 외래환자가 매일 복용 중이나 clinic visit 시 기록만 존재). 해당되면 SS=1, II=투여간격(hr)을 설정하는 dosing record를 만들어줘.
⏭ 모든 투여가 기록된 경우(입원 환자, Phase I 등) 건너뛰기.
→ Phase 3: Steady-state
Q14 특수 투여: CAR-T / 세포치료제 / 생물제제
COND해당시만
이 데이터가 세포치료제/CAR-T이면: AMT=주입 세포수(single infusion), 이후 세포 expansion은 DV(EVID=0)로 처리. DV가 투여 후 증가하는 패턴이 정상임을 인지하고, dosing record를 1회 infusion으로 설정해줘.
⏭ 일반 소분자/항체 약물이면 건너뛰기.
→ Phase 1: CAR-T 특수 사례
Q15 Dosing 데이터 중간 검증
MUSTAUTO
지금까지 만든 dosing 데이터프레임을 검증해줘:
✓ 모든 행에 EVID=1(또는 4), MDV=1인가?
✓ AMT > 0인가? (AMT=0인 행 없는지)
✓ 모든 ID에 최소 1개 dosing record 존재하는가?
✓ DATETIME에 결측이 있는가? 있으면 몇 건?
✓ ID별 투여 횟수 요약 통계 (min, median, max)
문제가 있으면 수정 코드도 함께 제공해줘.
→ Phase 6: Programmatic QC (dosing 부분)
1b
Observation Records 조립
EVID=0 — PK/PD 관찰값 구성
Q16–Q22
Q16 PK 관찰값 추출 & DV 설정
MUST
observation 소스에서 PK 농도 데이터를 추출하여 ID, DATETIME, DV, CMT, EVID=0, MDV, AMT=0 구조의 데이터프레임을 만들어줘. DV 단위를 확정하고 문서화해줘. PCSTRESN(숫자 표준화)과 PCORRES(원본)가 다른 행이 있으면 보여줘.
SDTM PC 도메인: PCSTRESN을 DV로 사용. PCSTAT="NOT DONE"인 건 제외 또는 MDV=1.
ADaM ADPC: AVAL을 DV로 사용.
→ Phase 1: Observation records 필수 변수
Q17 BLQ 식별 & 플래그 생성
MUST
정량한계 미만(BLQ) 데이터를 식별해줘. PCSTAT, PCORRES에서 "BLQ", "<LLOQ", "NS" 등의 텍스트를 찾아 BLQ 플래그(0/1)와 LLOQ 값 컬럼을 생성해줘. LLOQ 값은 어디서 가져올 수 있는지 확인해줘. BLQ 비율(전체, ID별, 시점별)도 요약해줘.
LLOQ 값: protocol의 bioanalytical section, 또는 PCLLOQ 변수, 또는 PC의 PCORRES에서 "<0.5" 같은 패턴에서 추출. BLQ 비율이 Phase 3의 M-method 선택에 직접 영향.
→ Phase 3: BLQ 처리 선행 작업
Q18 MDV 설정 로직
MUST
observation record의 MDV(Missing Dependent Variable)를 설정해줘:
• 유효한 관찰값 → MDV=0
• BLQ (M1 적용 시) → MDV=1
• 결측/미수행 → MDV=1
• BLQ (M3 적용 시) → MDV=0 (censored data로 likelihood에 기여)
현재 MDV 분포를 요약해줘.
→ Phase 1: MDV / Phase 5: EVID·MDV 조합
Q19 Observation CMT 할당
MUST
observation record에 CMT를 할당해줘. PK 농도 관찰이 central compartment에서 측정되므로 보통 CMT=2. PD endpoint가 있으면 별도 CMT 또는 DVID로 구분해줘.
→ Phase 1: CMT 기반 구분 / DVID
Q20 다중 분석물/다중 endpoint 처리
COND
Parent + metabolite 동시 모델링이나 PK+PD 통합 모델이면: 각 endpoint에 고유 CMT 또는 DVID를 할당하고, DV 단위가 다를 경우 scaling 필요 여부를 확인해줘. endpoint 간 magnitude 차이도 보고해줘.
⏭ 단일 분석물, 단일 endpoint이면 건너뛰기.
→ Phase 1: Multiple endpoint 처리
Q21 Predose sample 처리
MUST
Predose(투여 전) 채혈 record를 식별해줘. Single dose study에서 predose DV가 0이 아니면(detectable) 왜 그런지 표시해줘. Multiple dose에서 predose = trough이므로 정상적으로 DV>0일 수 있음. Predose sample의 TIME 처리 convention(TIME=0 vs TIME=-0.01)을 결정해줘.
→ Phase 3: Predose sample
Q22 Observation 데이터 중간 검증
MUSTAUTO
observation 데이터프레임을 검증해줘:
✓ 모든 행에 EVID=0인가?
✓ MDV=0인 행의 DV가 숫자(non-missing)인가?
✓ 모든 ID에 최소 1개 유효 observation(MDV=0) 존재하는가?
✓ BLQ flag와 MDV가 일관성 있는가?
✓ DV 범위 요약: min, Q1, median, Q3, max
✓ ID별 observation 수 요약 (rich vs sparse 판단)
→ Phase 6: Programmatic QC (observation 부분)

TIME 변수 — 가장 치명적인 단계
버그 #1 원인. 한 번 틀리면 전체 분석이 무의미
Q23–Q30
Q23 Reference time 정의
MUST
TIME 변수의 reference time(기준 시점)을 정의해줘. 표준은 각 ID의 첫 번째 투여 시점 = TIME 0. 이 reference datetime을 ID별로 추출하고, dosing 데이터에서 각 ID의 min(DATETIME)을 가져와서 REF_TIME 컬럼을 만들어줘.
ref_time <- dose_df %>% group_by(ID) %>% summarise(REF_TIME = min(DATETIME))
→ Phase 2: Reference time 선택
Q24 모든 DATETIME을 POSIXct로 파싱
MUST
dosing과 observation의 모든 날짜/시간 변수를 POSIXct로 통일 파싱해줘. lubridate::ymd_hms() 사용하되 timezone 인자를 명시적으로 설정해줘 (예: tz="UTC"). 파싱 실패 건수를 보고하고, 실패한 원본값 샘플을 보여줘.
Date class와 POSIXct를 혼합 연산하면 예기치 않은 결과 발생. 모든 것을 POSIXct로 통일.
